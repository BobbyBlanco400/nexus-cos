## üîê Critical Requirements

### 1. Device Identity Programming

**CRITICAL:** Each HoloSnap unit must have a **unique device ID** programmed during assembly.

**Format:** `HSNAP-[WAVE]-[SERIAL]`  
**Examples:**
- `HSNAP-W1-FTR-001` (Wave 1, Founder Tenant Rights holder #1)
- `HSNAP-W2-CRT-015` (Wave 2, Creator pre-order #15)

**Programming Method:**
- Flash device ID to EEPROM during firmware stub loading
- Include device ID in firmware metadata
- Ensure ID is immutable after programming

### 2. Inactive State Shipping

**CRITICAL:** All devices must ship in **inactive state**.

**Reason:** HoloSnap devices activate via **MetaTwin handshake** upon first user boot. This ensures:
- Canonical identity verification (Handshake 55-45-17)
- Tenant-aware activation (linked to pre-order ledger)
- Security and access control
- Lifecycle tracking from activation

**What This Means:**
- Firmware stub is pre-loaded but not fully activated
- Device powers on but displays "Awaiting Activation"
- Full functionality unlocks only after N3XUS authorizes

### 3. Quality Assurance Standards

**CRITICAL:** 98%+ pass rate target.

**Testing Requirements:**
- Power-on self-test (POST) ‚Äî 100% units
- Battery charging verification ‚Äî 100% units
- Wireless connectivity range test ‚Äî Sample basis (10%)
- Sensor calibration check ‚Äî 100% units
- Firmware stub validation ‚Äî 100% units

**Rework Process:**
- Any unit failing QA must be reworked or replaced
- Document all failures and resolutions
- Re-test after rework
